Industrial Ecology Digital Lab : Exiobase Server User Info

Created by Peter Esmé Maxwell, last modified on 21.04.2023

Summary

There is a virtual server, provisioned on NTNU's Openstack, solely for the purpose of EXIOBase. Actually, there are two servers.

  • EXIOSVR (exiosvr.iedl.no) : the main server for running EXIOBase pipelines, etc.
  • SECSVR (secsvr.iedl.no) : a smaller ancilliary server that handles various security functions and such like.


There are two types of access: unprivileged and administrator. This guide deals only with the former.

Warning: the files on EXIOSVR and SECSR are not backed-up, you must do this yourself!


Access and Login (non-admin accounts)

If you're been told you should have access then it'll have to be enabled. In other words, your access will need approved by Konstantin, and then either he or I (Peter) can actually enable it.

There is some background info on public keys for ssh here and general usage here.

SSH to EXIOSVR

This is fairly easy. You can use your NTNU username and password for this.

ssh username@exiobase.iedl.no

When prompted for your password, enter it, and you should have access. This may occasionally seem to pause for 10-20 seconds; this is when there hasn't been much activity and the cache is cold, nothing to worry about.

SSH to EXIOSVR using public key authentication

You can also ssh using public key authentication.

If public key authentication this is new to you, it means you generate a key pair on your local computer. The keypair has a private and a public part. You arrange to have the public part put on the systems you want to access, and only the matching private part of your key can then login against that public key. You can send your public key to anyone you like but NEVER send your private key.

So. You'll need to generate a public key and then use ssh with the password authentication to copy that into ~/.ssh/authorized_keys. It'll also need to have the correct permissions. This will then allow you to login using public key authentication, which if used with ssh-agent can make matters easier if you're copying files, etc.

Thankfully, this has been explained in our handy guide on ssh public keys here.


Accessing 'M' and 'X'

There are two standard shares setup that you can mount easily.

  • 'M' : //home.ansatt.ntnu.no/username (gets mounted to /home/adusers/username/mnt/M)
  • 'X' : //forskning.it.ntnu.no/ntnu/iv/ept/eksperimentell (gets mounted on /home/adusers/username/mnt/X)

To do this, there are some handy shell scripts in your home directory: mount_M.sh, mount_X.sh, umount_M.sh, and umount_X.sh.

If you want to mount the M share, just run ./mount_M.sh and it will mount it for you in ~/mnt/M. To unmount the share, just run umount_M.sh. Easy, huh?

Just remember to umount the shares before you finish your work as too many sessions hanging about can cause problems (if we see too many long-lived connections, your benevolent sysadmin may begin forcibly umounting).
