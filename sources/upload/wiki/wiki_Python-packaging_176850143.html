Providing a package is the preferred way to distribute Python code. Installing a Python package (through “pip install”) makes the code system-wide available; no need to fiddle around with path variables at the begin of script to get all modules importable.

This is a small tutorial which guides you through the steps necessary to provide your code as a python package. We use the country_converter (https://github.com/konstantinstadler/country_converter) as a minimal example.

You can find more detailed guides to python packaging here:

  • https://packaging.python.org/

  • https://the-hitchhikers-guide-to-packaging.readthedocs.io/en/latest/

Requirements

Minimal prerequisite:

  • A running Python installation, including pip - see Anaconda

Additional requirements for version control and code sharing

  • Git

  • A github or gitlab account

For an official python package

  • A pypi account (https://pypi.python.org/pypi?%3Aaction=register_form)

  • Twine (https://pypi.python.org/pypi/twine)

Package and project structure

Under the root folder of your package (some_folder/package_name) you need the folders

some_folder/package_name/package_name
some_folder/package_name/doc
some_folder/package_name/tests

The root folder

some_folder/package_name

The files in this folder describe your package and include all files necessary to install it:

  1. setup.py

  2. README (or README.rst / .md)

  3. LICENCE

  4. MANIFEST.in (the file MANIFEST is made automatically)

  5. requirements.txt

setup.py

This file describes your package and its requirements.

A minimal example would be:

from setuptools import setup
setup(
name='package_name',
description=(
'Some one line - '
'a short description of the package. '),
long_description=open('README.rst').read(),
url='path to github or gitlab',
author='your name',
author_email='your email',
version='0.0.1',
packages=['package_name', ],
license='GPLv3',
)

For a more complete example with console scripts and some additional data files see the setup.py of the country_converter.

README

A description of your project, in the best case with some snippets highlighting the functionality. This can be pure text or markdown/restructuredText format. For the later use README.rst as filename.

LICENCE

The licence of your package, you can download these from the internet after you decided on one. see https://choosealicense.com/ or the SI of Pauliuk, S., et al 2015 (10.1111/jiec.12316)

MANIFEST.in

These includes additional files you want to distribute with your package besides the source code. This should at least include

README.rst

if you use a specific file ending for your readme file.

requirements.txt

Any packages your package relies on with a version number, e.g:
pandas >= 0.17.0

The source code folder

some_folder/package_name/package_name

This includes your source code and must include a init.py

In the init.py you should have:

from package_name.package_name import *

author = 'your name'
version = '0.0.1'

You can specify which fucntionality should be visible to the user by modifiying the import statement. If you don't want to bother with the sync of the version number in here and in the setup.py, include as version.py and read the version number in init.py and setup.py (see country_converter for a working example).

The doc folder

For all additional documentation of your project, including tutorials, ipython notebooks, or a full sphinx documentation. See country_converter for a minimal example, a full documentation based on sphinx is used in pymrio.

The tests folder

All tests which should run to ensure the functionality of your project.

Packaging

Once you have everything in place run go to the root folder of your project and run

pip install .

This will install the package on your local computer. If it is already installed and you want to update run

pip install . -U

and if you want to remove it run

pip uninstall package_name

Distributing

You should have your package under version control and on a central repository. Push regularly to the central repo. Everyone cloning/downloading from there should be able to install in the same way as described above.

If you want to make a official package, go to the root folder of your package and run

python setup.py sdist

This will produce a folder dist at your projects (you should add this folder to your .gitignore). In this folder you will find your packaged project as a gz file. You can send this file around and other users can install it with

pip install package_name-version.tar.gz

To upload the package file to pypi (the official python package repository) use

twine upload package_name-version.tar.gz

This requires a .pypirc in your root with the following content

[distutils]
index-servers=pypi

[pypi]
repository = https://upload.pypi.org/legacy/
username = YOUR_NAME
password = YOUR_PW

Than your package should be available in pypi at
https://pypi.python.org/pypi/PACKAGE_NAME/VERSION

Note: The parsing of the readme at pypi is quite brittle, in particular for rst files. Any error in your readme will result in no parsing, in most cases I encountered so far, one of the internal links were not working.

Distributing to Anaconda

This guide assumes that you have a package already available on pypi and that you want to make it also availabe through the conda installer. If you want any other option (not make it open source but still want to install it through conda, check the conda user guide on https://conda.io/docs/user-guide/tutorials/index.html )

Steps

  1. Have a open pypi package (see pypi package guide above)

  2. setup Anaconda for uploading packages:

    1. get an anaconda account here https://anaconda.org

    2. conda install conda-build
    3. conda install conda-client
    4. anaconda login

  3. make a tmp directory and run

    1. conda skeleton pypi PACKAGENAME

    2. move to this directory and download build.sh and bld.bat to this directory (from https://conda.io/docs/user-guide/tutorials/build-pkgs-skeleton.html)

    3. if your package is platform / python version agnostic, add

      noarch: python

      to the section

      build:

    4. build the package for this directory by moving up one folder and run:

      conda-build PACKAGENAME

      at the end of the screen output you get the location of this file. You could upload to anaconda with the command given, e.g.

      anaconda upload /home/konstans/bin/anaconda3/conda-bld/linux-64/country_converter-0.5.4-py35hcc4c13d_0.tar.bz2

      or you set

      conda config -set anaconda_upload yes

      before

    5. If you need to specify architectures convert the package with

      conda convert -f -platform all DIR_FROM_BEFORE -o TMPFOLDER

      e.g.

      conda convert -f -platform all /home/konstans/bin/anaconda3/conda-bld/linux-64/country_converter-0.5.4-py35hcc4c13d_0.tar.bz2 -o ~/tmp/coco_all

      You have to manually upload these by switching to each folder and upload with anaconda upload FILENAME):

  4. check your upload on your anaconda dashboard

  5. install with

    conda install -c ACOUNTNAME PACKAGENAME

Automatic testing

If you have a tests directory, you can link your github account to travis and provide a

.travis.yml

file.

See also: https://travis-ci.org/getting_started

This will run automatic tests if you upload to github.

Similar things exist for gitlab, we are working on that.
