Goals

  • Streamline update procedure
  • Operate in a more recent classification (ISIC4 base)
  • Update to a more recent benchmark year (based on OECD ICIO)
  • Improve outcomes out of balancing routine (likely by tweaking approach and constraint coverage)
  • Have full country coverage in line with EXIOBASE3rx
  • Have flexiblity in extension coverage (reimplement characterisation matrices in a more professional way to allow for much higher degrees of freedom in running different extension sets).

Subgoals

  • Use APIs to gather raw data for EXIOBASE
  • Adopt EXIOBASE code to run parallel, potentially on external server/HPC
  • Improve/adopt now-casting
  • Improve data management (runs end in releases, which specific versions for publication)
  • Re-factoring of EXIOBASE code (unit tests) and transition to Python. Focus on modular structure

Background and strategic fit

EXIOBASE is a main asset of the Industrial Ecology Programme, necessary for research and to land new projects. However, EXIOBASE data becomes outdated and new data sources advancements of data access (e.g. API) can be used to streamline the compilation. Competition to other data providers also means that we must constantly innovate.

EXIOBASE code is currently hard to maintain, in particular to extend without knowledge of the whole code base. This makes is difficult to incorporate project work (e.g. from master students). A more modular and well documented structure promises to increase the number of contributors to the code.

Assumptions

  • EXIOBASE continues to be of use for research
  • We find PhDs/researchers which can take over the maintenance of EXIOBASE in the long run
