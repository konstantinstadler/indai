This is a set of conventions and rules for working at (and with) the IndEcol Digital Lab.

System and Languages

  • Linux is the preferred development system but aim for compatibility with Windows.

  • All projects/modules/functions should be available and run on the IndEcol Server.

  • If a program is developed in Python, aim for package which can be accessed via command line and Matlab.

Use available resources

  • Do not reinvent the wheel.
    Before starting a new project or coding, spend some time investigating if the functionality is available, either within IndEcol or as a public module.

Project structure

  • Every project must have a gitlab or github page.

  • A readme is mandatory, a tutorial is nice to have.

  • If you program in python, aim for package as the final product.

Documentation

  • Every function/module/procedure/source code file must have a summary line as first line or immediately following the function declaration (independent of programming language and length of code)

  • Aim for self-explaining code through modularization and function/variable names to keep comment at a minimum and in sync with the code.

  • Do not comment the what but the why.

Python

Use numpy docstyle format for all docstrings and the Black code formater for automatic formatting of all code (use the standard settings).

Communication

  • All project specific communication takes place at through the issue tracker on the specific project.
    See also http://www.yegor256.com/2016/08/23/communication-maturity.html

  • General tips and tricks are gathered in the IndEcol Digital Lab wiki.

  • IEDL Slack for ad hoc discussions

Required reading

These are must reads:

  • Wilson et al 2014: Best Practices for Scientific Computing. DOI: 10.1371/journal.pbio.1001745

  • Pauliuk et al 2015: Lifting Industrial Ecology Modeling to a New Level of Quality and Transparency: A Call for More Transparent Publications and a Collaborative Open Source Software Framework. DOI: 10.1111/jiec.12316

  • Wilson et al 2017: Good enough practices in scientific computing. DOI: 10.1371/journal.pcbi.1005510

Additional (optional) reading:

  • http://www.paulgraham.com/head.html

  • Rougier et al 2014: Ten Simple Rules for Better Figures. DOI: 10.1371/journal.pcbi.1003833

  • http://book.openingscience.org/
